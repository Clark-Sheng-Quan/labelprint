# Copilot 工作规范

## 核心原则

### 1. 禁止创建文档文件
- **规则**：不要创建任何 `.md` 文档、总结或说明文件
- **例外**：除非明确要求创建项目文档
- **原因**：保持代码库整洁，信息通过对话交流
- **违反后果**：删除所有非必要的 md 文件

### 2. 重大改动需预先通知
- **规则**：根据用户的语气判断是否需要确认
  - **纯疑问/问题**（如"为什么..."、"怎么..."、"是不是..."）→ 先回答和分析，**等待用户确认**后再改
  - **疑问 + 要求**（如"为什么...改成..."、"为什么...删除..."）→ 先简要分析，**直接执行改动**
  - **明确要求**（如"删除..."、"改成..."、"可以"、"开始"）→ 直接执行改动
  - **特大改动**（预计 30+ 分钟或涉及 3+ 文件）→ **必须先通知**，不管什么语气

- **触发条件（需要提前通知）**：
  - 涉及 3 个以上文件的改动
  - 预计耗时 30 分钟以上
  - 改变现有功能的实现逻辑
  - 重构代码结构或架构
  - **所有 Java 文件改动**（即使只涉及 1-2 个文件）

- **执行流程**：
  1. 分析用户语气
  2. 纯疑问 → 先回答分析，等待确认
  3. 疑问+要求 或 明确要求 → 直接执行（小/中改动）或先通知（大改动）
  4. 特大改动 → 必须先通知确认
  5. 不要未经许可就开始大规模修改

## 工作流程

### 3. 代码注释和日志语言
- **规则**：所有代码注释、日志消息、错误提示必须用英文
- **范围**：适用于所有新增代码（TypeScript、Java、Kotlin 等）
- **包括**：
  - 函数/类的注释
  - 代码行注释
  - Log.d() / Log.e() 等日志消息
  - 错误提示字符串
  - 异常消息
- **不包括**：代码变量名、函数名等（保持已有命名约定）

### 4. 配置管理规则
- **禁止规则**：不要在代码文件中硬编码配置的默认值
- **问题代码**（❌ 错误）：
  ```javascript
  const dbConfig = {
    host: process.env.DB_HOST || 'localhost',      // 硬编码默认值
    port: process.env.DB_PORT || 5432,             // 硬编码默认值
    database: process.env.DB_NAME || 'miketea_machine', // 硬编码默认值
  };
  ```
- **正确做法**（✅ 正确）：
  - 在 `.env` 文件中定义所有值（包括默认值）
  - 代码中只读取环境变量，不提供备用值
  - 或使用专门的 `config/` 目录管理所有默认配置
  ```javascript
  const dbConfig = {
    host: process.env.DB_HOST,
    port: process.env.DB_PORT,
    database: process.env.DB_NAME,
  };
  ```
- **原因**：
  - 避免代码和配置文件中配置重复
  - 配置变更时只需改一个地方
  - 防止代码中的默认值与实际配置不一致

## 常见场景

| 场景 | 处理方式 |
|-----|--------|
| 修复单个文件的 bug | 直接修复 ✅ |
| 添加简单功能到单个组件 | 直接添加 ✅ |
| 修改 Java 文件（任何改动） | 先通知 🛑 |
| 修改 3+ 个 React/TypeScript 文件 | 先通知 🛑 |
| 重构订单流程 | 先通知 🛑 |
| 调整 UI 样式（单文件） | 直接修改 ✅ |
| 修改 API 调用方式 | 先通知 🛑 |
| 测试代码修改 | 直接修改 ✅ |
| 重构状态管理系统 | 先通知 🛑 |
